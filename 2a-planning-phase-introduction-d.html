Process code for software procurement

2a. Planning phase (name?)

The first phase was about discovery, and it culminated in choosing a strategy for obtaining software – an existing solution (proprietary or open source) or a new one (building in-house or contracting a vendor). You’re here because you chose to build software in-house.
This second phase is about planning. You’ll articulate a plan and performance criteria that guide your development process.
At the end of Phase 2, you will have a solid plan for the software development process.

Building software

Creating software is an exciting opportunity to shape a tool that will help you and your colleagues do your job better. The work you have done so far (from User Journeys to Total Cost of Ownership analyses) will help you create an agile development plan. That plan will help you reduce risk, while ensuring a high-quality end product. You will describe a full software package that addresses the original problem statement, but also break the work into smaller, interoperable modules that can be built individually, and continuously deployed and integrated.
Agile software development is vast topic. If your team has capacity to build in-house, you already know more than this guide can teach you. Click through to reconnect with the procurement path for guidelines on implementation, transitioning your code to an open source license, and maintenance.
